# Architecture Rule 11: Security

## Overview

Security concerns in the application center on **token storage**, **token refresh**, **secure transmission**, and **sensitive data handling**. Authentication tokens are stored in encrypted storage, refreshed transparently, and never exposed in logs or error messages.

## Token Storage

Tokens are stored using `flutter_secure_storage`, which uses the platform keychain (iOS) or EncryptedSharedPreferences (Android).

### ITokenStorage Interface

```dart
abstract interface class ITokenStorage {
  Future<void> saveTokens({
    required String accessToken,
    required String refreshToken,
  });
  Future<String?> getAccessToken();
  Future<String?> getRefreshToken();
  Future<void> clearTokens();
}
```

### SecureTokenStorage Implementation

```dart
class SecureTokenStorage implements ITokenStorage {
  const SecureTokenStorage(this._storage);
  final FlutterSecureStorage _storage;

  static const String _accessTokenKey = 'access_token';
  static const String _refreshTokenKey = 'refresh_token';

  @override
  Future<void> saveTokens({
    required String accessToken,
    required String refreshToken,
  }) async {
    await Future.wait([
      _storage.write(key: _accessTokenKey, value: accessToken),
      _storage.write(key: _refreshTokenKey, value: refreshToken),
    ]);
  }

  // ... getAccessToken, getRefreshToken, clearTokens
}
```

### Provider

```dart
@riverpod
ITokenStorage tokenStorage(TokenStorageRef ref) {
  final secureStorage = ref.watch(secureStorageProvider);
  return SecureTokenStorage(secureStorage);
}
```

## Token Refresh Flow

The `RefreshTokenInterceptor` handles automatic token refresh:

1. A request returns 401 Unauthorized.
2. The interceptor acquires the queue lock (`QueuedInterceptor`), blocking concurrent requests.
3. It calls the `/auth/refresh` endpoint using a **separate Dio instance** (no auth interceptors) to avoid recursion.
4. On success: saves new tokens, retries the original request with the new token, releases the queue.
5. On failure: clears all tokens, calls `onAuthExpired` callback (which invalidates the Dio provider, triggering auth state re-evaluation), rejects all queued requests.

```
Normal Request
     |
     401 Unauthorized
     |
RefreshTokenInterceptor (QueuedInterceptor)
     |
     +-- Read refresh token from secure storage
     |
     +-- POST /auth/refresh (using plain Dio, no interceptors)
     |       |                    |
     |    200 OK              401/Error
     |       |                    |
     |   Save new tokens     Clear all tokens
     |       |                    |
     |   Retry original       Call onAuthExpired
     |   request                  |
     |       |                Redirect to login
     |   Return response
```

## Riverpod Bridge

Dio interceptors operate outside Riverpod's provider tree. The bridge between them:

```dart
RefreshTokenInterceptor(
  tokenStorage: tokenStorage,
  dio: refreshDio,
  onAuthExpired: () {
    // Invalidate the Dio provider, which cascades to auth state
    ref.invalidateSelf();
  },
),
```

## Sensitive Data Handling

### Logging Redaction

The `LoggingInterceptor` must redact sensitive headers:

- `Authorization` headers are logged as `Bearer [REDACTED]`.
- Request bodies for auth endpoints are not logged.
- Response bodies containing tokens are not logged.

### Error Messages

- `Failure.message` contains developer-friendly messages, not user-facing text.
- User-facing error messages are generated by `failure_message_mapper.dart` using localized strings.
- Stack traces are included in failures for debugging but never displayed to users.

### Config Files

- `config/development.json`, `staging.json`, `production.json` may contain sensitive values (Sentry DSN).
- Consider adding production config to `.gitignore` if it contains secrets.
- Never store API keys, passwords, or client secrets in config files -- use CI/CD environment variables.

## Storage Rules

| Data type | Storage | Rationale |
|-----------|---------|-----------|
| Access token | `flutter_secure_storage` | Encrypted, short-lived |
| Refresh token | `flutter_secure_storage` | Encrypted, long-lived |
| Theme preference | `SharedPreferences` | Not sensitive |
| Locale preference | `SharedPreferences` | Not sensitive |
| Feature flags | `SharedPreferences` | Not sensitive |
| User profile cache | `SharedPreferences` | Low sensitivity, convenience |

## DO

- Store all authentication tokens in `flutter_secure_storage`.
- Use the `ITokenStorage` interface, never access `FlutterSecureStorage` directly from features.
- Clear tokens on logout, regardless of whether the server-side logout succeeds.
- Use a separate Dio instance for the refresh endpoint to prevent interceptor loops.
- Redact sensitive data in all log output.
- Use HTTPS for all API communication in staging and production.

## DO NOT

- Do not store tokens in `SharedPreferences` -- it is unencrypted.
- Do not log full Authorization headers, tokens, or passwords.
- Do not include tokens or credentials in error messages or failure objects.
- Do not hardcode API keys, secrets, or tokens in source code.
- Do not disable SSL/TLS verification in production (only acceptable in development for proxy debugging).
- Do not expose the `FlutterSecureStorage` instance directly to feature code -- always go through `ITokenStorage`.
