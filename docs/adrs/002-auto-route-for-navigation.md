# ADR 002: auto_route for Navigation

## Status

Accepted

## Context

Flutter's built-in `Navigator 2.0` API is powerful but verbose. The application needs:

- Type-safe route definitions with compile-time parameter checking.
- Route guards for authentication-based access control.
- Nested navigation for tabbed layouts (bottom nav, navigation rail, drawer).
- Deep linking support for mobile and web.
- Integration with Riverpod for reading auth state in guards.

Alternatives considered:

- **go_router**: Official Flutter team recommendation. Good deep linking support but limited nested navigation ergonomics, less powerful guard system, and route definitions are string-based rather than type-safe.
- **Navigator 2.0 directly**: Maximum flexibility but extreme verbosity. Router delegate and information parser boilerplate is substantial.
- **Beamer**: Declarative approach but smaller community and less active maintenance.

## Decision

Use **auto_route** with **auto_route_generator** for all navigation.

Key implementation choices:

- The `AppRouter` is created inside a `@Riverpod(keepAlive: true)` provider so that `Ref` can be passed to route guards.
- `AuthGuard` extends `AutoRouteGuard` and reads `isAuthenticatedProvider` via `ref.read()` to synchronously check authentication.
- Authenticated routes are wrapped in an `AutoTabsRoute` shell that provides the adaptive scaffold (bottom nav, rail, or drawer depending on screen size).
- Route names are generated by stripping the `Page` suffix from widget class names (configured via `replaceInRouteName: 'Page|Screen,Route'`).

## Consequences

### Positive

- **Type-safe navigation**: `router.push(ProfileRoute(id: userId))` is checked at compile time.
- **Guard composability**: Multiple guards can be stacked on routes or route groups.
- **Shell routes**: `AutoTabsRouter` integrates cleanly with the adaptive scaffold pattern for responsive navigation.
- **Deep linking**: Path-based routing works out of the box with `MaterialApp.router`.

### Negative

- **Code generation**: Another generator in the `build_runner` pipeline (`.gr.dart` files).
- **Learning curve**: auto_route has its own concepts (route pages, shell routes, guards) that differ from vanilla Navigator.
- **Riverpod bridging**: Passing `Ref` to `AuthGuard` via the router provider is a non-obvious pattern that must be documented.

### Neutral

- All pages must be annotated with `@RoutePage()` and routes defined in `AppRouter.routes`.
- Route changes require running `build_runner` to regenerate the `.gr.dart` file.
